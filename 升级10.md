0. 文档定位
本文档不是讨论稿，也不是愿景说明书。本文档是给 Codex / 工程代理直接执行的升级规格书，目标是：
1. 把 Alpha-Insight 当前已暴露的全部主要痛点收敛成可执行任务。
2. 明确不能动什么、必须补什么、先后顺序是什么。
3. 保证每一阶段都具备 DoD、验证方式、证据输出、回滚路径。
4. 在不破坏 Telegram 主链路前提下，完成系统统一化、实时化、可观测化、可回归化。
本文档采用 Upgrade9 的工程执行骨架，吸收 Upgrade7 的强制修改清单与协议底座，补入 Upgrade8 的产品体验目标，并追加前端实时化升级补充章节。
1. 背景与目标
Alpha-Insight 当前已经具备 Telegram 请求入口、统一研究链路、行情与新闻抓取、指标生成、监控与告警、治理与降级、双前端、证据文件与回归门禁雏形。
但仓库公开文档同时表明，当前系统仍存在几个关键收口问题：Next 控制台仍依赖离线 JSON 导出、配置分散、观测面板不完整、合规文档缺失、端到端 smoke 不足、Postgres 与 SQLite 存在错位、Webhook 对端协议只有最小约定、生产托管策略未给出。
升级10的目标不是继续加功能，而是解决系统已经长大但统一层没打平的问题。
2. 本次升级的总目标
一句话目标：把 Alpha-Insight 从功能很多的复杂原型，升级为统一状态源、统一契约、统一入口、统一观测、统一回归门禁的可持续演进系统。
本次必须解决的结果包括：前端不再依赖离线 JSON 作为主数据源；Web 与 Telegram 语义一致；主编排职责拆层；运行时协议标准化；Store 与 evidence 统一；运维路径固定；smoke / regression gate 完整；旧入口可灰度回滚。
3. 非目标
1. 不新增新的大业务模块。
2. 不推翻现有 Telegram 主链路。
3. 不重做整个研究逻辑。
4. 不先做 UI 美化，再补底层。
5. 不在未冻结 contract 之前大规模改前端。
4. 证据基础与事实边界
本规格书只基于仓库中当前公开可见的事实编写：README 中明确给出关键链路、证据目录、Roadmap、TODO、Unknown / Needs verify；Upgrade7 明确给出当前核心缺口、To-Be 原则、强制修改清单、P0 证据要求；Upgrade8 明确把问题焦点定义为用户可感知的产品层升级；Upgrade9 明确给出 Console API / BFF、typed contract、双轨运行、P0/P1/P2、测试门禁与执行约束。
凡未被仓库公开文档确认的内容，本文档一律不当作已实现事实。
5. 全量痛点清单（按严重度排序）
P0-A：系统统一性问题。痛点1：主数据链路没有统一真相源。README 明确写出 docker-compose.telegram.yml 拉起 Postgres，但当前 Python 存储实现是 SQLite；同时 Next 控制台读取的是导出的 upgrade7_frontend_resources.json，而不是直连主存储。后果是运行层、展示层、部署层可能分别依赖不同状态源，排障和一致性验证成本高。
痛点2：前端主链路仍是离线快照，而不是实时控制台。README 明确写出 Next 控制台当前读取 docs/evidence/upgrade7_frontend_resources.json，需要先跑导出脚本。Roadmap 也明确把升级到实时 typed API / OpenAPI contract 列为第一项。后果是前端更像资源浏览器，不像真正的控制台。
痛点3：Web 与 Telegram 仍缺统一语义契约。Upgrade9 专门写了 Web 与 Telegram 语义一致硬规则，这说明这件事当前仍然是待建设目标，而不是天然已经完全解决。后果是 Web 触发和 Telegram 命令可能结果相近但不完全同构，导致用户理解和测试覆盖都不稳定。
P0-B：核心运行时与协议问题。痛点4：fallback 路径安全一致性仍不足。Upgrade7 直接点名 fallback 路径安全一致性仍可提升，Policy 与 Runtime 未完全解耦，存在后门风险。
痛点5：ExecutionResult 未彻底成为唯一执行结果标准。Upgrade7 把 ExecutionResult 标准化列为 P0 必须项，并要求 Debugger / 报告只读标准字段。后果是执行链条不同环节仍可能依赖 runtime 私有字段，后续重构和替换 runtime 会很痛。
痛点6：ToolResult 输出协议仍未全域统一。Upgrade7 直接写了 Tool 输出协议不统一，汇总层存在多套字段兼容逻辑；并将 ToolResult 标准化列入 P0。后果是汇总层、UI 层、报告层都在做额外适配，容易漂。
痛点7：Graph / Node 契约测试不够硬。Upgrade7 明确指出节点契约测试与集成回归分层可加强，升级迭代风险偏高，并把 Node I/O Contract 固化列入 P0。
痛点8：RunStore / ArtifactStore 未固化。Upgrade7 直接写明 RunStore / ArtifactStore 未固化，会限制 run 回放、审计、Web 控制台演进。
P0-C：可扩展性与职责边界问题。痛点9：策略 / 告警 / 风控逻辑仍偏内嵌，插件化不足。Upgrade7 明确点名这类逻辑仍偏内嵌业务逻辑，插件化不足，扩展成本高。
痛点10：前端缺 typed contract 解析与守卫层。Upgrade9 特别提出在 web_console/lib 建解析器与类型守卫，保证字段缺失 / 新增时页面可降级渲染。
痛点11：能力映射矩阵还没被冻结。Upgrade9 把旧版能力到新版页面一一映射矩阵列为 P0。
痛点12：运行入口过多，标准路径不够清晰。README 列出 Telegram 链路、调度、脚本、Streamlit、Next.js、资源导出等多种入口和壳层。
P1：观测、运维、协议文档问题。痛点13：配置文档分散。README Roadmap 直接写出 Telegram 与 runtime config 键当前分散在脚本与代码，需要形成单一配置手册。
痛点14：观测面板还不够直观。README Roadmap 明确写出需要把 metric_events / degradation_events 直接可视化，减少人工查库。
痛点15：端到端 smoke 不完整。README Roadmap 写明还需补 webhook + scheduler + pulse 的可重复 smoke。Upgrade9 也要求测试门禁覆盖 Telegram 与 Console。
痛点16：webhook 对端协议文档不足。README Unknown 直接指出 WebhookTextSender 仅有最小约定 {target, text}，缺正式接口文档。
痛点17：生产托管策略未正式化。README Unknown 直接指出 systemd / supervisor / k8s 未在仓库给出。
痛点18：合规文档不完整。README 写了已有安全边界，也明确指出爬虫 / RSS 的 robots 与站点条款合规策略未看到统一策略文件，并建议补 docs/compliance*.md 或白名单策略。
P1：产品与体验一致性问题。痛点19：用户首屏体验与底层能力还未完全闭环。Upgrade8 明确把当前瓶颈定义为不是系统能不能跑，而是用户第一眼是否得到答案、是否愿意持续使用。
痛点20：失败与恢复反馈需要统一模板。Upgrade9 要求所有动作必须提供：已受理、执行结果、失败原因、下一步建议。
6. 升级原则
1. 状态单源：统一 store 是唯一真相源。
2. 协议优先：先冻结 contract，再动实现。
3. 前后端解耦：前端只能依赖 API / BFF + typed contract，不允许直接吃内部结构。
4. 双轨运行：新入口灰度，旧入口保留可回滚。
5. 观测先行：新增能力必须带 audit / metrics / evidence。
6. 不破坏 Telegram 主链路：任何重构不得影响现有命令可用性。
7. 目标架构（To-Be）
目标架构要求：Web 和 Telegram 通过不同入口进入，但最终调用同一 action 能力；所有运行结果、告警、治理状态、证据都落同一主存储；前端只读取 typed API，不直接依赖内部 JSON / SQLite 细节；Evidence 保留，但从前端主数据源降级为验收与回归证据层。
建议链路：Telegram / Web Console 进入 Unified Gateway / Action Layer；BFF 进入 Runtime Orchestrator；Orchestrator 与 Tool Adapters、Sandbox Runtime、Reliability Governor 协作；最终落到 Single Source Store；前端再通过 Typed Resource API 读取 Runs、Alerts、Governance、Evidence。
8. 强制约束（Codex 必须遵守）
1. 不允许删除现有 Telegram 命令链路。
2. 不允许直接让前端读取内部 store 表结构。
3. 不允许继续扩散新的输出 schema。
4. 不允许在 contract 未冻结前重写页面。
5. 不允许跳过测试门禁直接改主入口。
6. 不允许把运行时私有字段暴露给用户界面。
7. 不允许以先跑通为由绕过 policy/runtime 一致性。
9. 分阶段执行计划
Phase 0：契约冻结与能力映射。目标是先把系统到底长什么样冻结住，避免边改边漂。必须完成能力映射矩阵、统一 typed contract、资源 API 与前端契约冻结、状态枚举冻结。建议落点文件包括 web_console/lib/contracts.ts、web_console/lib/parsers.ts、web_console/lib/type_guards.ts、core/contracts.py、core/execution_result.py、core/tool_result.py、docs/upgrade10_capability_matrix.md。DoD：能力映射矩阵覆盖率 100%，typed contract 有 schema version，前后端 contract 对齐，所有状态枚举集中定义。
Phase 1：统一数据源与前端实时化。目标是把前端从离线资源视图升级成实时控制台。必须完成单一主数据层、前端改用实时 API、typed BFF、前后端字段去漂移。建议落点 services/resource_api.py、services/store_adapter.py、services/run_store.py、services/artifact_store.py、web_console/app/api/*、web_console/lib/contracts.ts、scripts/upgrade7_frontend_resources_export.py（降级为兼容工具）。DoD：前端能不依赖导出 JSON 运行；Runs、Alerts、Governance 页面可实时显示主数据；字段新增 / 缺失不会导致前端白屏。
Phase 2：运行时协议与编排拆层。目标是把当前大编排文件 + 多套兼容逻辑的风险拆掉。必须完成 Policy / Runtime 分层、ExecutionResult 全链标准化、ToolResult 全链标准化、Node I/O Contract 测试、workflow_engine 拆层。建议落点 core/sandbox_policy.py、core/sandbox_runtime.py、core/execution_result.py、core/tool_result.py、agents/workflow_graph.py、agents/workflow_nodes.py、agents/workflow_result_builder.py、agents/workflow_governance_hooks.py。
Phase 3：统一入口、插件化、治理面板。目标是把很多功能散着能用变成统一入口、统一治理、统一审计。必须完成 Telegram / Web 统一 Action 层、策略 / 告警 / 风控插件化、治理面板实时可视化、用户反馈模板统一。建议落点 services/action_service.py、services/governance_read_model.py、services/plugins/*.py、web_console/app/(ops)/*、web_console/components/governance/*。
Phase 4：运维、协议文档、回归门禁。目标是让系统不是能跑，而是能长期稳定运营。必须完成统一配置手册、webhook contract 正式化、生产托管模板、合规文档、smoke + regression gate。建议落点 docs/configuration_manual.md、docs/webhook_contract.md、docs/compliance.md、deploy/systemd/* 或 deploy/compose/*、tests/smoke/test_webhook_smoke.py、tests/smoke/test_scheduler_smoke.py、tests/smoke/test_market_pulse_smoke.py。
10. 用户输出契约
为了避免 Codex 在 UI / 文案上随意发挥，统一如下：
10.1 动作反馈统一模板：所有 Web / Telegram 动作必须提供四段——已受理、执行结果、失败原因、下一步建议。
10.2 默认视图去噪：默认视图隐藏 run_id / request_id / schema / internal keys；Debug 视图可看内部字段。
10.3 首屏原则：结果先给结论，再给证据和按钮。
11. 测试与质量门禁
11.1 必跑门禁：pytest -q；Telegram 回归 tests/test_telegram_phase_b.py、tests/test_telegram_phase_d.py、tests/test_market_pulse.py；Console 的 npm run typecheck 与 npm run build；Web action 与 Telegram command 输出一致性测试；关键字段 strategy_tier、tier_guarded、degrade 渲染测试。
11.2 新增必须测试：contract tests、store consistency tests、realtime API tests、webhook contract tests、smoke tests、rollback tests。
12. 回滚与双轨策略
新能力必须双轨灰度：8501 / 8502 / 8503 旧入口持续运行；8600 或新入口增量覆盖；严重回归可立即切回旧版，不影响在线服务。
13. Codex 执行规则
1. 每一 Phase 单独提交。
2. 每一 Phase 都先补 contract / test，再补实现。
3. 先打平底层协议，再做前端页面覆盖。
4. 每阶段输出：变更文件清单、设计摘要、DoD 自查、测试结果、证据 JSON。

14. 最终验收标准
项目可以视为本次升级完成，当且仅当同时满足：前端不再依赖 upgrade7_frontend_resources.json 作为主数据源；Web / Telegram 动作同参同构；ExecutionResult / ToolResult / Node Contract 全域统一；RunStore / ArtifactStore 固化，可支撑回放与审计；治理面板可视化 metric / degradation / recover；webhook / scheduler / pulse smoke 可重复执行；生产托管、配置手册、合规文档齐备；Telegram 主链路在整个升级过程中未被破坏。
15. 一句话总结给 Codex
先冻结契约，再统一数据源；先保 Telegram 不坏，再让 Web 真正接管；先把协议、Store、治理、门禁打平，再谈继续长功能。
16. 前端实时化升级补充章节（Realtime Console Addendum）
16.1 现状判断：当前前端还不是同类产品意义上的实时控制台。README 已明确说明，Next 控制台当前读取的是 docs/evidence/upgrade7_frontend_resources.json，并且在启动 Next.js 控制台前，需要先执行 python scripts/upgrade7_frontend_resources_export.py。这说明现在的主链路仍然是先导出资源快照，再给前端展示，而不是前端直接读取在线状态。README 的 Roadmap 已经把前端升级目标写得很明确：Next 控制台数据链路需要从离线 JSON 导出升级到实时 typed API / OpenAPI contract。
16.2 能否做到同类产品的实时更新：能做到，但要分层实现。从仓库现有设计目标看，Upgrade9 已要求 Web 与 Telegram 语义一致，monitor 任务对象与 Telegram /monitor 同构，strategy_tier、tier_guarded、降级 / 恢复状态与 Telegram 路径同源，并提供统一时间线和状态回显。这套目标本质上就是一个实时感很强的运营控制台目标。它差的不是页面设计，而是数据通道：当前仍以资源快照为主，而不是实时 API / 事件流。
16.3 实时化目标分级：A. 准实时（本次必须达成）——页面按 2 到 5 秒刷新一次关键资源，用户主观感受为状态在持续更新。适用页面包括 /runs、/alerts、/governance、/monitors、最近事件时间线。B. 事件实时（本次建议预留）——关键事件到达后，前端在秒级内更新，不必等待完整页面轮询，适用事件包括 run 状态变化、alert 新增 / 关闭、degrade / recover 变化、delivery 成功 / 失败变化。C. 强实时（本次不强制）——多人协作、日志流、毫秒级刷新、强交互同步等高级场景。
16.4 本次推荐方案：采用两段式实时化方案。先做 Realtime API + SWR / 短轮询，再做关键事件流补强。原因是：它与 README 已确认的目标一致；它与 Upgrade9 的 Console API / BFF + typed contract + 双轨灰度完全一致；它不会过早把系统拖进高复杂度的实时通信架构。
16.5 强制执行要求：一，不允许继续把资源导出当成主数据源。upgrade7_frontend_resources_export.py 和 upgrade7_frontend_resources.json 只能保留为兼容导出工具、验收证据快照、回归比对样本，不允许继续作为 8600 前端主入口的数据源。二，前端必须只读 BFF / API。Web Console 不允许直接耦合 SQLite、内部 store 表结构或脚本导出文件；必须通过 Console API / BFF 访问数据。三，所有实时页面必须走 typed contract。所有页面都必须通过解析器与类型守卫转换后再渲染；后端字段缺失、新增或版本升级时，页面只能降级，不允许白屏。四，实时化不得破坏 Telegram 主链路。8600 的前端实时化只能增量灰度，旧入口 8501 / 8502 / 8503 保持可回退。
16.6 目标接口定义：本次实时化至少新增或正式化以下 API：GET /api/runs、GET /api/runs/:id、GET /api/alerts、GET /api/governance、GET /api/monitors、GET /api/events?since=...。其中 /api/runs 用于 runs 列表、状态刷新、搜索筛选；/api/alerts 用于 Alert Hub；/api/governance 用于降级、恢复、DLQ、投递成功率等状态卡；/api/monitors 用于监控列表与状态；/api/events 用于底部时间线。
16.7 页面刷新策略：列表页 /runs、/alerts、/monitors、/governance 默认 3 秒轮询，页面失焦降为 10 秒，页面隐藏时暂停，用户手动刷新始终可用。详情页 /runs/:id、/alerts/:id 以 1 到 2 秒轮询状态字段，证据区按需刷新，详情完成后自动降频。时间线优先改为事件流或单独高频刷新。
16.8 事件流补强策略：本次建议预留事件流接口，但不要求一开始覆盖全部页面。推荐优先级第一层为 run status changed、alert created、alert resolved、degrade started、recover finished、delivery failed；第二层为 metric_events 聚合变化、scheduler tick、pulse generated。
16.9 代码落点建议：后端包括 services/resource_api.py、services/governance_read_model.py、services/events_read_model.py、services/store_adapter.py、web_console/app/api/*；前端包括 web_console/lib/contracts.ts、web_console/lib/type_guards.ts、web_console/lib/parsers.ts、web_console/lib/polling.ts、web_console/lib/realtime.ts、web_console/app/(dashboard)/runs/page.tsx、web_console/app/(dashboard)/alerts/page.tsx、web_console/app/(dashboard)/governance/page.tsx。

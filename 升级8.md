# Alpha-Insight Upgrade8：产品体验与24x7分析升级（Telegram优先）
**基线截至：2026-02-28**

> 目标：在 Upgrade7 工程底座上，完成“用户可感知”的产品层升级。  
> 原则：先让用户在首屏看懂“涨跌结论”，再给原因、证据与动作；降低刷屏与操作负担，提升对话自然度与可信度。  
> 体验共识：同一屏信息必须同时满足“可读、可验证、可行动”（结论 -> 证据 -> 按钮）。

---

## 0. 背景与定位

Upgrade7 解决了“后端可维护、可治理、可扩展”的底层问题。  
现阶段主要瓶颈已从“系统能不能跑”转为“用户第一眼是否得到答案、是否愿意持续使用”。

Upgrade8 聚焦 Telegram 产品体验，形成“结论先行 + 可验证证据 + 低噪声交互 + 24x7持续价值”的闭环。

对标主流行情与终端产品（过滤优先、噪声治理、并排信息对照）后，Upgrade8 在不推翻 Upgrade7 底座的前提下，补齐“首屏回答、选择效率、交互节奏、可验证证据”四个关键短板。

---

## 1. 升级目标（2026 Q2）

### 1.1 产品目标
- 首屏直接回答用户问题（先给涨跌，再给原因）。
- 降低交互噪声（进度消息单条更新，结果单次发出）。
- 新闻信息从“数量展示”升级到“主题与影响”。
- 歧义标的从“阻断等待”升级为“可理解、可快速确认”的选择器。
- 首屏按钮收敛，减少选择困难。
- 统一“结论先行 + 可操作入口 + 降噪”的首屏体验。
- 引入 LLM 自然对话能力：自然问法可执行，低置信度先澄清再执行。

### 1.2 体验目标
- 同一请求最多两条核心消息：`进度消息` + `最终结果卡片`。
- 默认不展示内部运行字段（run_id/request_id），仅在调试模式可见。
- 用户在 3 秒内看到“已受理/处理中”，在结果消息中直接看到涨跌结论。

### 1.3 运营目标
- 引入 24x7 运行状态可见化（`/status` 已具备，升级8完善展示口径）。
- 形成“主动价值触达”能力：在监控链路外补充市场脉冲摘要（可灰度）。
- 扩展新闻与公告来源覆盖（官方披露 + 市场资讯 + 宏观日历），降低单一来源偏差。

---

## 2. 非目标（防止范围失控）

- 不在 Upgrade8 内重写 Upgrade7 的底层架构（GraphState/Policy/Connector 不重做）。
- 不引入自动交易执行能力（依旧坚持 research/alert 范畴）。
- 不做全渠道一致改造，优先 Telegram 完整闭环，再复用到其他渠道。
- 不在本阶段改造策略引擎核心逻辑，重点在输出组织与交互节奏。

---

## 3. 当前问题诊断（面向用户体感）

### 3.1 首屏回答不足
- 用户问“一个月涨跌”，系统首屏常先给技术模板而非涨跌结论。
- 涨跌额、区间位置、振幅等高价值字段未标准化呈现。

### 3.2 歧义选择体验生硬
- 命中多个候选时阻断流程，且选择策略不透明。

### 3.3 消息噪声高
- 同 request 可能出现多条阶段性与重复分析消息，造成“不稳定”感。

### 3.4 新闻价值密度低
- “近7天X条”偏日志化，不足以支持快速判断。

### 3.5 证据对用户不友好
- 证据字段偏内部口径，缺少“来源、更新时间、指标口径”这类可验证信息。

## 4. Upgrade8 信息架构（First Screen Contract）

## 4.1 首屏卡片结构（固定4段）

### Card A｜区间表现（必须回答用户问题）
- 近30天涨跌幅(%) + 涨跌额(价格差)
- 区间高/低、振幅、最大回撤
- 当前价在区间位置（高位/中位/低位）

### Card B｜原因摘要（技术 + 新闻各1句）
- 技术一句话：方向判断 + 触发条件（RSI保留但不抢主位）
- 新闻一句话：主题Top3 + 是否为价格驱动因素判断

### Card C｜证据三件套（可验证，不日志化）
- 行情源 + 更新时间（到分钟/收盘）
- 新闻源覆盖 + 抓取时间窗
- 指标口径（RSI14(1d)、均线周期等）

### Card D｜动作入口（按钮收敛）
- `📈K线` `📰新闻` `🔔提醒` `更多`

## 4.2 技术一句话标准
- 结构：`方向判断 + 触发条件 + 风险说明`
- 示例：
  - `短线偏弱，若站回10日均线并放量可转中性；否则维持震荡偏弱。`

## 4.3 新闻模块标准
- 不再只报条数，改为：
  - 主题 Top3（财报/监管/产品/宏观/其他）
  - 每主题一句影响摘要
  - 每主题1条代表新闻（标题+时间+来源）
- 可选增强：主题热度/变化率（用于市场脉冲场景，不强制首屏展示）。

## 4.4 首屏按钮标准
- 首屏仅保留：`📈K线` `📰新闻` `🔔提醒` `更多`
- 其余按钮（报告/重试/近3个月/只看新闻）收进“更多”。

## 5. 交互协议升级（Telegram Conversation Contract v3）

## 5.1 消息节奏
- 每个请求只允许：
  - 1条进度消息（编辑更新）
  - 1条最终结果卡片
- 禁止同 request_id 发送重复最终卡片。
- 落地依据：Telegram 支持编辑已发送消息（`editMessageText` 类能力），可实现“单条进度消息持续更新”。
- 硬规则（可测试）：
  - `progress_message_count_per_request == 1`
  - `final_message_count_per_request == 1`
  - `final_message_send` 对 `(request_id, final_schema_version)` 幂等

## 5.2 去噪规则
- 默认隐藏：`run_id/request_id/schema/action/internal metric keys`。
- 内部证据进入 audit/store/evidence，不进主消息。

## 5.3 歧义选择器
- 候选展示标准：`代码 + 市场 + 公司名`（最多5条）。
- 支持自然语言确认：`选港股那个` / `选美股`。
- 默认选择策略可配置：
  - 中文语境默认 HK
  - 或按用户历史偏好优先
- 增加“过滤优先”入口：`港股/美股/A股/加密` 快速筛选按钮（按场景可裁剪）。
- 候选排序透明化：`语言/地区默认 + 历史偏好 + 流动性（成交额/市值）兜底`。

## 5.4 LLM 自然对话协议（NLU/NLG）
- 优先解析自然语言意图，不直接回命令模板。
- 意图解析输出统一动作协议，由现有 policy/guardrails/route strategy/strategy_tier 执行，不绕过安全约束。
- 低置信度时先进行一次关键槽位澄清（标的/市场/周期），再落地执行。
- 对话类请求（如“你能做什么”“你好”）返回中文能力卡与示例，不返回内部技术字段。
- 所有自然对话输出同样受去噪规则约束（不暴露 run_id/request_id/schema/action/internal keys）。

## 6. 24x7 分析能力增强（在现有调度基础上增量）

## 6.1 运行状态卡（Status Card）
- 统一口径展示（用户可读5行）：
  - 监控数（active）
  - 24h 投递成功率 / 失败重试
  - 当前数据源状态（正常/降级/缺失）
  - 最近投递延迟（last_delivery_latency）
  - 最近一次异常与恢复时间（可选）
- `/status` 作为用户可观测入口，保持简洁与可解释。
- 当降级发生时，必须对用户可解释（降级原因 + 影响范围 + 恢复状态）。

## 6.2 主动脉冲摘要（可灰度）
- 新增 `市场脉冲` 任务（建议每1h或4h）：
  - Top movers（涨跌幅）
  - 新闻主题变化/热度变化
  - 风险提示
- 仅在用户开启订阅后主动推送，默认关闭。

## 8. 路线图（P0 / P1 / P2）

### P0（止血，必须）
1. 首屏补齐涨跌/区间/位置字段，并按“结论优先”排序。
2. 同 request_id 结果去重；进度消息改“编辑同一条”。
3. 收敛首屏按钮到 4 个主入口。
4. 证据块改为用户可验证三件套（来源、更新时间、口径）。
5. 体验指标：`首屏命中率`（问涨跌时包含涨跌/区间/位置）>= 95%。
6. 数据源指标：优先接入 `HKEX RSS + SEC RSS + Investing RSS`，并在 `/status` 暴露源健康。

### P1（核心体验升级）
1. 歧义选择器升级（候选标准化 + 自然语言确认 + 默认策略透明）。
2. 新闻主题化输出（Top3主题 + 每主题一句影响 + 代表新闻）。
3. 技术一句话改“方向+触发条件”。
4. 体验指标：`歧义确认平均操作步数 <= 1.5`。
5. 体验指标：`自然语言意图识别成功率 >= 90%`，`低置信度直接拒绝率 <= 5%`。
6. 数据源指标：接入 `PR Newswire / Business Wire / Seeking Alpha`（标题+摘要+链接）。

### P2（运营增强）
1. 市场脉冲摘要（订阅制主动推送）。
2. Status Card 增强（历史趋势、异常提示、恢复状态）。
3. 多渠道复用 Telegram 输出契约。
4. 体验指标：主动推送退订率 < X%，投诉率 < Y%。
5. 数据源指标：接入 `FRED + Fed release schedule + NY Fed calendar` 形成宏观预告脉冲。

## 11. 风险与回滚策略

### 11.1 风险
- 首屏字段增加导致消息超长。
- 进度编辑与并发消息冲突。
- 歧义默认策略误选带来理解偏差。

### 11.2 回滚
- 通过 feature flag 回退至 Upgrade7 输出模板。
- 关闭进度编辑，回退为静态单条“处理中”。
- 关闭市场脉冲主动推送，仅保留手动查询。

## 12. 完成定义（Definition of Done）

- 用户问“一个月涨跌”，首屏必含涨跌结论与区间位置。
- 每请求最多两条核心消息（进度1 + 结果1）。
- 默认不暴露内部字段（run/request/schema/action）。
- 新闻区可读性提升：主题化 + 影响摘要 + 来源可追溯。
- `/status` 与能力卡可一致呈现“7x24运行能力”。
- 自然问法可直达可执行动作，低置信度优先澄清而非直接拒绝。

## 13. 与 Upgrade7 的关系

- Upgrade7 是“工程底座升级”。
- Upgrade8 是“产品可感知升级”。

两者关系：
- Upgrade7 提供可治理与可观测能力，保证 Upgrade8 能稳定落地。
- Upgrade8 把 Upgrade7 的能力转化为用户看得懂、愿意持续使用的体验价值。

## 14. LLM 自然对话升级（NLU/NLG）

### 14.1 目标
- 把“命令驱动”升级为“自然问法优先”，减少用户学习成本。
- 让机器人在低置信度场景先澄清、再执行，降低误拒绝与误执行。

### 14.2 能力范围
- 覆盖核心意图：分析、新闻、监控、状态、路由策略、帮助问答。
- 支持多轮澄清：标的歧义、周期缺失、市场未明确。
- 对话兜底：寒暄、能力询问、操作指引统一中文回复。

### 14.3 执行边界
- LLM 仅负责 NLU/NLG（理解与表达），不直接调用高风险执行路径。
- 所有执行请求仍走 Upgrade7 统一策略链：policy/guardrails/route/tier。
- 对外输出保持“可读、可验证、可行动”，默认不暴露内部运行字段。

### 14.4 验收指标
- `intent_success_rate >= 90%`
- `low_confidence_hard_reject_rate <= 5%`
- `clarification_to_success_rate >= 85%`
- `template_fallback_rate` 持续下降（以自然问法样本统计）

## 15. 数据源扩展计划（与当前项目切合度评估）

### 15.1 与现有代码的切合度结论
- 结论：**高切合，可增量接入**。当前已具备 RSS 连接器、统一重试/限流语义、`news_digest` 统一字段与 Telegram 证据块，符合“先扩源、后增强”的实施路径。
- 现状依据（代码层）：
  - `tools/news_data.py` 已支持多 feed 并行抓取、统一解析为 `{title, link, published_at, summary, source}`。
  - `core/connectors.py` 已具备标准化错误语义（429/5xx/超时/解析失败）与重试退避。
  - `services/news_digest.py` 已支持 Top5、事件聚类、情绪方向、来源覆盖统计。
- 约束点：
  - SEC 场景需补充 `symbol -> CIK` 映射，才能稳定做公司级 filings 过滤。
  - 新闻稿/研报类需严格遵守 ToS，展示层采用“标题+摘要+链接”。

### 15.2 来源分层与接入优先级
- `P0（收益最高）`：官方披露 + 高覆盖资讯
  - HKEX RSS（港交所官方）：https://www.hkex.com.hk/Services/RSS-Feeds?sc_lang=en
  - HKEX RSS 中文页：https://www.hkex.com.hk/Services/RSS-Feeds?sc_lang=zh-HK
  - SEC RSS（含 EDGAR 入口）：https://www.sec.gov/about/rss-feeds
  - Investing.com RSS：https://www.investing.com/webmaster-tools/rss
- `P1（公司动态与研报增强）`
  - Nasdaq RSS：https://www.nasdaq.com/nasdaq-rss-feeds
  - PR Newswire RSS：https://www.prnewswire.com/rss/
  - PR Newswire Asia RSS：https://www.prnasia.com/rss/
  - Business Wire feed options：https://www.businesswire.com/help/feed-options
  - Seeking Alpha feeds：https://about.seekingalpha.com/feeds
- `P2（宏观脉冲与日历）`
  - FRED Release Calendar：https://fred.stlouisfed.org/releases/calendar
  - Federal Reserve Release Schedule：https://www.federalreserve.gov/data/releaseschedule.htm
  - NY Fed Economic Indicators Calendar：https://www.newyorkfed.org/research/calendars/nationalecon_cal

### 15.3 爬虫与合规契约（强制）
- 优先使用官方 RSS/日历入口；避免高频抓网页正文。
- 遵守 robots.txt 与 ToS；新闻稿/研报类默认不落全文。
- 用户可见层仅展示：`标题 + 时间 + 来源 + 摘要 + 跳转链接`。
- 证据层记录：`source_id/feed_url/fetch_at/http_status/parse_ok/item_count`。

### 15.4 加入后的收益（预期）
- 可信度提升：官方源（HKEX/SEC/Fed）显著提升“证据三件套”的权威性。
- 覆盖度提升：公司事件、监管动态、宏观发布三类信号覆盖更完整，减少“仅有条数、缺少内容”。
- 稳定性提升：RSS 结构稳定、增量抓取成本低，与现有 Connector 治理模型匹配。
- 解释性提升：Top3 主题与来源覆盖更充分，降低“分析像日志”的观感。
- 运营效率提升：`/status` 可直接反映各源健康，便于灰度扩源与故障回退。

升级12：收口定版建议稿
Upgrade12 Closure｜面向彻底收口、统一口径、长期运行
文档定位

•	本次升级不是继续扩功能，而是补齐最后一公里，把主干已成的系统收口成正式版本。
•	核心目标是让前后端资源层、实时体验、文档口径、运维/合规/门禁三者一致。
•	本稿延续升级10/11格式，但聚焦“彻底修复”和“定版”。

1. 当前状态判断
1.1 已具备的成熟基础
•	统一资源 API 已成为 Next 控制台主链路，覆盖 runs、alerts、governance、monitors、events、evidence。
•	前端 typed contract、parser、type guard、polling、realtime 已进入正式结构。
•	旧的 upgrade7_frontend_resources_export.py 已降级为证据/回归 fallback，不再是控制台主依赖。
•	存储口径已经收敛为 SQLite 主链路，Postgres 仅为未来迁移预留。
•	webhook_contract、compliance、deploy/systemd 等外围文档已存在。
1.2 仍未彻底收口的问题
•	monitors 前端链路已落地，但仍需补“回归门禁 + 文档口径”收口，避免后续回退。
•	当前落地的是 polling/realtime 准实时链路，关键事件推送式实时还未正式写成已完成能力。
•	资源 contract 已有 schema version，但每类资源的独立版本治理和兼容策略尚未体系化。
•	compliance 文档仍有 remaining verify items，制度层尚未完全闭环。
•	systemd 文档更偏部署基线，还不是完整运行手册。

2. 升级12总目标
一句话目标：把 Alpha-Insight 从“主干已成的平台骨架”，升级为“前后端资源一致、关键事件实时、运行说明完整、门禁可长期执行”的收口定版。
2.1 本次必须完成的结果
1.	monitors 资源链路回归门禁与文档口径锁定。
2.	关键事件通道从纯轮询升级为“轮询 + 事件流补强”。
3.	资源 contract 版本治理细化。
4.	compliance remaining verify items 关闭。
5.	deploy/systemd 从基线升级为运行手册。
6.	配置治理信号清理完毕。
3. 非目标
•	不新增新的大业务模块。
•	不重写研究核心编排。
•	不切换主存储到 Postgres。
•	不一次性把全站改成 WebSocket。
•	不为了炫技引入超出当前阶段的复杂中间件。
4. 升级12核心问题清单
问题 1：前端资源链路需要“防回退门禁”
monitors 已在前端资源面落地，但缺少专门防回退门禁，后续迭代容易出现“文档写有、页面退化”的隐性回归。
问题 2：实时化仍以 polling 为主
当前 polling.ts + realtime.ts 已实现高频刷新、失焦降频、隐藏暂停与手动刷新，但当前主能力还是准实时刷新，不是关键事件推送式实时。
问题 3：资源版本治理还不够细
虽然资源 API 已进入正式 schema 资源层，但每类资源 contract version / compatibility policy 尚未被独立写清。
问题 4：compliance 仍有尾项
合规边界虽然不再空白，但尚未完全定版。
问题 5：systemd 文档还是部署基线，不是完整运行手册
当前更偏“怎么装”，而不是“怎么长期运营”。
问题 6：仓库治理信号还不够干净
配置示例与真实配置边界还可以更明确。
5. 升级12原则
7.	先收口，再扩展。
8.	前端资源面必须与后端资源面一致。
9.	关键事件优先做轻量事件流，不做全站重型实时。
10.	文档必须落后于实现的问题一次性修完。
11.	运行文档、合规文档、回归门禁一并定版。
6. 执行方案（分阶段）
Phase A：前端中文化与资源门禁补强
目标：完成 8600 控制台中文/双语化，同时锁定 monitors 等关键资源链路不回退。
必做任务
•	8600 全页面中文化（允许双语），包含导航、统计卡、筛选、表头、提示文案。
•	补 monitors 专项回归门禁，防止资源页面回退。
•	README 与升级文档统一说明 monitors 已完成，不再表述为待补。
DoD
•	8600 页面不再出现纯英文业务文案。
•	monitors 回归门禁通过。
•	Web 与 Telegram 的监控列表对象同构。
•	监控列表页面可实时显示监控任务。
验收
前端中文/双语页面可显示 monitor 列表、状态、下次执行、策略层级，并与 Telegram /list 结果语义一致。
Phase B：关键事件实时化补强
目标：在保留现有 polling 主链路的基础上，给最关键的状态变化补事件流。
必做任务
•	为 events 资源新增 SSE 通道。
•	为 alerts 状态变化新增 SSE 通道。
•	前端时间线页面支持 EventSource 订阅。
•	列表页仍保留 polling，不全站强制改成推送。
DoD
•	时间线能在秒级看到关键事件新增。
•	告警状态变化无需等待完整列表轮询。
•	断开 SSE 时可自动回退到 polling。
验收
第一批事件建议覆盖 degrade_started、recover_finished、guard_triggered、delivery_failed、alert_created、alert_resolved。
Phase C：资源 contract 版本治理
目标：把现有 typed contract 从“能用”升级为“长期可维护”。
必做任务
•	为每类资源定义独立 contract version。
•	增加 compatibility policy 文档。
•	定义 breaking / non-breaking 变更规则。
•	将 contract 变更纳入 smoke / regression gate。
DoD
•	前后端能明确知道某类资源的契约版本。
•	契约升级不会默默打爆页面。
•	兼容策略有文档、有测试。
验收
建议资源粒度：runs、alerts、governance、monitors、events、evidence 各自独立版本。
Phase D：制度层收尾
目标：把还剩待确认的外围治理项全部关掉。
必做任务
•	关闭 compliance.md 里的 remaining verify items。
•	明确 retention period policy。
•	明确 Telegram source IP / outbound webhook target 的生产 allowlist 口径。
•	为 webhook contract 增加版本记录区。
•	给 systemd 文档补日志轮转、备份恢复、staging/prod 差异、健康检查矩阵。
DoD
•	compliance 不再残留待确认项。
•	deploy/systemd 文档不再只是安装说明。
•	运维人员可按文档长期维护系统。
验收
制度层文档要变成 closed 状态，而不是 roadmap 状态。
Phase E：仓库治理与配置清理
目标：让仓库的专业信号更干净。
必做任务
•	根目录仅保留 .env.example。
•	README 明确 .env 不进入版本控制。
•	部署文档统一从 .env.example 生成运行配置。
•	检查 scripts / deploy / docs 中是否仍引用真实 .env。
DoD
•	配置示例与真实配置边界清晰。
•	新人不会把本地配置当仓库正式文件。
验收
这一步不影响功能，但直接影响项目成熟度信号。

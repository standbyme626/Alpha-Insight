0) 先定义“完成得如何”的评分输出（你跑完就能填）

每次跑完一轮，输出一个 run_report.jsonl（或表格）最少包含：

run_id

mode（planner/full/fused/report）

symbol/market/date_range

data_source

sandbox_backend（docker/e2b/fallback）

network_mode（none/allowed）

retry_count

success

latency_ms

failure_type（data/sandbox/parse/model/unknown）

artifacts（html/png/pdf/json 等数量）

metrics_provenance_ok（关键数值是否来自 stdout/metrics）

这就是阶段 C 的“可观测”最小闭环，同时也是阶段 A/B 的验收证据。

1) 阶段 A：真沙箱可用性（P0）怎么测
A1. 断网容器回归（核心）

目标：--network none 下 Full/Report 仍能成功（证明“容器外取数、容器内计算”真的完成了）。

测试用例

同一个标的（比如 AAPL / 00700 / 600519）各跑 5 次

运行模式：Full Analysis（Week2）+ Report（Week3，如果有）

Docker 强制 --network none

期望：

success=true

stderr 不出现网络请求失败（如 yfinance 请求、DNS 失败等）

sandbox_backend=docker:quant-sandbox（或你 UI 显示的 docker backend）

验收统计

连续 20 次成功率 ≥ 95%

docker backend 占比 ≥ 95%

fallback 触发次数 ≤ 1（最好 0）

A2. “禁止容器内拉数”静态扫描

目标：从源头防止回退到“脚本里拉 yfinance”。

做法

grep/静态规则：在沙箱脚本模板或 coder 输出里禁止：

yfinance.download、requests.get、http(s)://、socket、aiohttp

如果你有 guardrails，就跑 guardrails 单测（见门禁部分）

2) 阶段 B：统一 Full + Fused（单请求单报告）怎么测
B1. run_id 一致性测试

目标：一次请求产生一个 run_id，三个 Tab 都引用它。

测试用例

发起一次请求（例如“分析 AAPL：技术指标+新闻情绪+结论”）

读取 UI/后端返回对象：

plan.run_id == full.run_id == fused.run_id

三个 Tab 不再“跨运行拼接”

验收

10 次随机请求，run_id 一致率 100%

强校验触发时能明确报错（而不是 warning 静默）

B2. 统一结果 schema 完整性测试

目标：ResearchResult 或你定义的结果对象字段齐全。

最小必测字段

request

plan

data_bundle_ref / data_source

sandbox_backend

metrics（结构化）

artifacts_index（图表/报告索引）

provenance（数值引用指针）

retry_count、errors（如有）

验收

schema 校验 100% 通过（建议 pydantic 或 jsonschema）

3) 阶段 C：可观测与运营怎么测
C1. 指标埋点“是否真的有”

目标：至少能按天/按 backend 看：

success rate

fallback rate

avg latency

retry rate

failure_type 分布

测试方法（最小版）

连跑 50 个任务（混合市场/标的）

输出一份聚合报告：

overall + by backend + by mode + by market

验收

每个指标都有数值（不是空/None）

failure_type 能分类（不是全部 unknown）

C2. 告警阈值回归（如果你已做）

目标：fallback 连续触发/失败率突增能报警。

测试

人为让 docker 不可用（或模拟）

看是否触发阈值并产出告警记录

4) 质量门禁（持续执行）怎么测

最硬的就是把你计划书第 6 节落成 CI 命令组合：

python -m py_compile 覆盖新增/修改文件

pytest -q 全绿

至少 1 次真实 docker 沙箱运行验证（记录 backend 证据）

UI 8502 可访问且无 StreamlitAPIException

（如果你有）guardrails：危险导入/网络调用/路径越界拦截的单测

5) 给你一套“最短验证路径”（半天内就能拿到硬证据）

断网容器 Full Analysis 20 连跑（最关键，证明 Alpha-Insight-78）

单请求生成三 Tab，run_id 一致 10 次（证明统一输出协议）

50 次混合任务跑完输出指标聚合（证明可观测）

pytest -q + py_compile（证明门禁）

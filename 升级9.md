# Alpha-Insight Upgrade9：统一控制台与全能力对齐升级（Web Console 优先）
**基线截至：2026-02-28**

> 目标：在 Upgrade8 体验契约稳定的基础上，完成“新版前端（Next.js）覆盖旧版前端与 Telegram 可用能力”的产品化升级。  
> 原则：能力先对齐，再做美化；语义先一致，再做交互；全程保留旧版入口，双轨灰度。  
> 共识：任何在 Telegram 可执行的动作，必须在新版前端可见、可配、可追踪（且语义一致）。

---

## 0. 背景与定位

Upgrade8 完成了 Telegram 优先的产品可感知升级，解决了“首屏读不懂、证据不可验证、交互噪声高”的核心问题。  
当前瓶颈已转向“能力分散在多入口，Web Console 覆盖不全，运维与分析操作割裂”。

Upgrade9 的定位是：把现有能力统一为“一个前端主入口 + 一套能力契约 + 多入口并存可回退”的运营化体系。  
旧版 Streamlit（8501/8502/8503）继续保留，作为并行入口与应急回退路径。

---

## 1. 升级目标（2026 Q2~Q3）

### 1.1 产品目标
- 新版前端覆盖旧版前端全部核心能力（研究、驾驶舱、治理、证据）。
- Telegram 可执行能力在新版前端全量集成（监控、路由、偏好、批量、告警中心等）。
- 保持 Upgrade8 关键契约不回退：`route strategy / strategy tier / degrade / 去噪`。
- 形成统一“能力面板 + 执行入口 + 审计追踪”闭环。

### 1.2 体验目标
- 一个主入口（8600）即可完成：分析、监控、治理、配置、追溯。
- 关键动作（如 monitor/route/webhook/pref）必须提供表单化与结果回显，不依赖命令记忆。
- 结果页默认展示用户可读字段，内部字段只在 Debug/运维面板显示。

### 1.3 运营目标
- 运维操作从“命令行/Telegram 分散执行”升级为“可视化执行 + 可追溯日志”。
- 关键状态可观测：降级状态、恢复状态、告警交付质量、策略层级命中。
- 维持双轨运行：新版灰度期间旧版持续可用，故障可一键回退。

---

## 2. 非目标（防止范围失控）

- 不在 Upgrade9 重写 Upgrade7/Upgrade8 的核心工作流引擎。
- 不引入自动交易执行能力（仍为 research/alert/ops 范畴）。
- 不要求一次性下线旧版 Streamlit 页面（先并行，后切换）。
- 不改变 Telegram 命令语义与安全约束（前端仅做等价可视化封装）。

---

## 3. 当前问题诊断（面向全链路）

### 3.1 能力入口割裂
- 8501 偏扫描告警，8502 偏研究，8503 偏资源治理，8600 当前仅部分覆盖。
- 用户/运维需要在多个页面或命令入口之间切换，操作成本高。

### 3.2 能力语义未统一前端化
- Telegram 命令可执行的动作，并未全部映射为 Web Console 可操作组件。
- `strategy_tier`、`tier_guarded`、degrade 事件在前端的可见性不足或不连续。

### 3.3 数据契约边界不清
- 前端读取方式存在快照与资源 API 混用，容易出现字段漂移与页面回归。
- 缺乏统一 typed contract 与版本化策略。

### 3.4 运维追踪链路不完整
- 路由配置、偏好配置、批量操作、webhook 等动作缺少统一审计视图。
- 告警失败与降级恢复缺少单页时间线串联。

## 4. Upgrade9 信息架构（Capability Contract）

## 4.1 顶层导航（主入口 8600）

- `Research`：规划 + 统一研究 + 历史 + 产物
- `Cockpit`：扫描、信号、流水线、告警预览与触达
- `Telegram Ops`：命令能力可视化配置与执行
- `Alerts`：告警中心（状态/失败/抑制/层级）
- `Governance`：降级/恢复状态与时间线
- `Evidence`：证据文件、门禁结果、运行快照
- `Runs`：运行记录与关键指标对照

## 4.2 Telegram Ops 子模块（必须对齐命令）

- `Status` 对应 `/status`
- `Monitors` 对应 `/monitor /list /stop`
- `Reports` 对应 `/report /digest`
- `Alert Hub` 对应 `/alerts`
- `Bulk Actions` 对应 `/bulk`
- `Webhook` 对应 `/webhook`
- `Routes` 对应 `/route`
- `Preferences` 对应 `/pref`

## 4.3 数据与执行双面板

- 左侧：配置/筛选/执行入口
- 右侧：执行结果、状态回显、错误提示、审计摘要
- 底部：最近事件时间线（degrade/guard/failure/recover）

---

## 5. 统一交互协议（Web ↔ Telegram 语义一致）

## 5.1 语义一致性硬规则

- Web 触发 monitor 与 Telegram `/monitor` 生成同构任务对象。
- Web 展示的 `strategy_tier` 与 Telegram 路径完全一致。
- Web 显示的 `tier_guarded` 与抑制原因与 Telegram 告警中心一致。
- Web 的“降级中/恢复”状态与 `/status` 同源。

## 5.2 去噪规则一致

- 默认视图隐藏内部字段（run_id/request_id/schema/internal keys）。
- Debug 视图可查看内部字段，用于审计与排障。
- 对外提示先给可执行建议，不直接暴露内部异常栈。

## 5.3 失败与恢复反馈

- 所有动作必须提供：`已受理`、`执行结果`、`失败原因`、`下一步建议`。
- 失败统一映射错误码与用户可读文案，保留技术细节到审计层。

---

## 6. 技术架构增量（不破坏现有链路）

## 6.1 Console API / BFF 层（新增）

- 统一封装前端读写接口，避免前端直接耦合内部存储。
- 所有写操作复用 `services/telegram_actions.py` 与 `agents/telegram_command_router.py` 的既有能力。
- API 输出走 typed contract，支持 schema version。

## 6.2 前端契约层（新增）

- 在 `web_console/lib` 建立解析器与类型守卫：
  - 输入：后端 JSON/快照
  - 输出：稳定的前端类型对象
- 保证字段缺失/新增时页面可降级渲染而不崩溃。

## 6.3 双轨运行机制（保留旧版）

- 8501/8502/8503 持续运行。
- 8600 增量覆盖能力并灰度验证。
- 任何严重回归可切回旧版入口，不影响在线服务。

---

## 7. 路线图（P0 / P1 / P2）

### P0（能力映射与契约冻结，必须）
1. 输出“旧版能力 -> 新版页面”一一映射矩阵（无遗漏）。
2. 冻结前端 typed contract（含 `strategy_tier/tier_guarded/degrade`）。
3. 统一资源 API 与快照读取策略，消除字段漂移。
4. 验收：映射覆盖率 100%，关键字段前端可见且可筛选。

### P1（Telegram 能力全量前端化）
1. 上线 Telegram Ops 模块（status/monitor/report/digest/alerts/bulk/webhook/route/pref）。
2. 动作执行结果可回显、可重试、可审计。
3. 与命令链路一致性测试（同参数同结果）。
4. 验收：Telegram 可用功能在前端可执行覆盖率 >= 95%。

### P2（驾驶舱与研究全融合）
1. 把 8501/8502 的核心能力集成到 8600（Research + Cockpit）。
2. 提供统一运行时间线（研究 -> 告警 -> 交付 -> 治理）。
3. 增强可视化：筛选、聚合卡、错误/降级时间线。
4. 验收：主流程（分析->监控->告警->报告）可在 8600 单入口完成。

## 10. 指标口径（建议）

- 功能覆盖率：旧版能力在 8600 映射覆盖率
- 可执行覆盖率：Telegram 可用功能的前端可执行比例
- 一致性通过率：Web 与 Telegram 同动作同结果比例
- 运维效率：关键任务平均操作步数、平均定位时长
- 回归质量：门禁通过率、线上回滚次数

## 11. 风险与回滚策略

### 11.1 风险
- 前端一次性并入过多能力导致交互复杂度上升。
- 前后端契约变更引发历史页面回归。
- Web 与 Telegram 参数映射不一致导致行为偏差。

### 11.2 回滚
- 保持 8501/8502/8503 持续可用（主回滚策略）。
- 8600 功能按模块 feature flag 灰度开启。
- 写操作走后端幂等校验，失败可安全重试。

---

## 12. 完成定义（Definition of Done）

- 用户在 8600 单入口可完成核心流程：分析、监控、告警治理、证据追溯。
- Telegram 可执行功能在 8600 均有对应操作入口与结果回显。
- `route strategy / strategy tier / degrade / 去噪` 不回退且可视化可验证。
- 旧版入口保留，可在升级期作为稳定回退路径。
- 回归门禁与证据文件完整，支持复盘与交接。

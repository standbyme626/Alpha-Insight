1) 主流产品的“投研业务流程”共性（联网归纳）

把主流产品拆开看，它们的流程大致会落在同一个闭环：

发现机会：Screener/Scan/Watchlist（批量筛选）→ 形成候选池

TradingView 强调 watchlist 与 watchlist alerts 来跨多个标的监控条件触发。

持续监控：自定义 alerts / feed / 推送，避免盯盘

Bloomberg 强调可定制化 alerts + news/data/research 一体化覆盖，用于持续跟踪假设与变化。

深度检索与证据：把“非结构化内容”变成可引用证据（文档/电话会/研报/新闻）

AlphaSense 强调生成式搜索把定性洞见与结构化金融数据连接，并支持统一工作流与可审计输出。

研究计算/回测验证：Research → Backtest/Live（研究产物进入可复现环境）

QuantConnect 明确 Research Environment（训练/分析）与 Backtest/Live 的衔接（可保存模型/对象，导入回测结果再分析）。

输出与复用：研报/图表/结论 + 可追溯证据 + 可复现计算

Bloomberg/AlphaSense 都强调“统一平台”“可审计/可追溯输出”的方向。

3) Alpha-Insight 升级计划书（对齐你现状 + 吸收主流流程）
0. 计划目标（2026 Q1~Q2）

把 Alpha-Insight 从“能跑的多链路 Demo”升级成“单请求单报告、可追溯、可运营”的投研闭环系统：

业务闭环：发现→监控→研究→验证→交付→复盘

技术底座：数据/计算解耦 + 沙箱真隔离 + 统一结果协议 + 指标运营

1. 现状对齐（As-Is 摘要）

双入口：8501 驾驶舱；8502 LLM 控制台

8502 三模式：Planner / Full Analysis（沙箱+自修复）/ Market+News（融合但不走沙箱）

P0 痛点：断网沙箱里拉 yfinance 导致“真沙箱但数据失败”（Alpha-Insight-78）
（以上均来自你给的“现状重构版”，这里不再展开）

2. To-Be 总体架构原则

数据获取在沙箱外；沙箱只做计算/绘图/回测（对齐断网隔离目标）

单请求 → 单 run_id → 单“研究结果对象”（UI 仍可多 Tab）

证据与审计：每个关键数值必须能追溯到 sandbox stdout / metrics（对齐主流“可审计输出”方向）

监控与告警是产品核心（对齐 TradingView/Bloomberg 的 alerts 思路）

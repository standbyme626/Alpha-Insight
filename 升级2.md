1) 主流产品的“Telegram投研/监控”流程共性（联网归纳，截至 2026-02-26）

把“在 Telegram 里分析 + 监控 + 推送”拆开看，市场上主流做法基本是同一套四层架构：

接入层（Bot Inbound）：  
使用 Telegram Bot API 的 `getUpdates`（轮询）或 `setWebhook`（回调）接收用户命令，二者互斥；未消费更新在服务器端最多保留 24 小时。  

策略层（Command Router + Orchestrator）：  
把 `/analyze`、`/monitor`、`/list`、`/stop` 等命令解析成标准任务，请求统一研究引擎或扫描引擎执行。  

数据层（Market Data Stream + Snapshot）：  
生产系统通常采用流式数据（WebSocket）做实时触发，避免高频轮询；再配合分钟/小时级聚合用于稳定告警。  

通知层（Push + State）：  
所有触发与推送都写入状态存储（订阅、最近触发、去重键、失败重试），确保“可追溯、可恢复、可审计”。

典型产品映射（公开资料）：
- TradingView：支持价格/技术/Watchlist Alerts，并可通过 Webhook 推送到外部自动化系统。  
- TrendSpider：支持 Alerts Webhook，支持变量模板、超时与固定来源 IP。  
- Koyfin：支持价格/技术/估值/新闻告警，覆盖 Watchlist 和多端通知。  
- CoinMarketCap TG Bot：直接在 Telegram 里查询价格、开启小时级提醒（命令驱动）。  
- Freqtrade（开源）：Telegram 命令控制（start/stop/status/profit），强调“命令集 + 状态可视化”。

结论：  
“Telegram 作为交互面，研究/监控引擎在后台运行”已经是成熟模式，不是新概念；差异点在于数据质量、调度稳定性、幂等与审计链路。

```mermaid
  A[Telegram 用户命令<br/>/analyze /monitor /list /stop] --> B[Bot Gateway<br/>Webhook/Long Polling]
  B --> B1[update_id 落库与去重<br/>unique(update_id)]

  D -->|查询/停用| G[订阅管理器]

  F --> H[调度器 APScheduler/Celery]
  H --> I[scan_watchlist / symbol scan]
  I --> J{触发条件命中?}

  T --> L[结果摘要器]

  M --> M1{发送成功?}
  M1 -->|是| N[审计日志与快照<br/>notifications + snapshots + run_id]
  M1 -->|否| RQ[retry queue]
  RQ --> M
  RQ --> DLQ[dead-letter]

  N --> O[观测指标<br/>success/failure/retry/latency]
```

3) Alpha-Insight 升级计划书（生产版 Telegram 接口 + 24h 监控）

0. 计划目标（2026 Q1-Q2）

把 Alpha-Insight 升级为可生产运行的 Telegram 量化投研助手：
- 在 Telegram 内完成“命令式分析 + 监控管理 + 定时推送”全流程。
- 支持 7x24 任务调度、异常恢复、重复消息防抖、审计留痕。
- 保持现有 Full Analysis 与 Watchlist 能力复用，减少重构风险。

1. 需求合理性分析（为什么现在做）

业务合理性：
- 用户入口收敛：从“网页触发”扩展为“即时消息触发”，显著降低使用门槛。
- 价值闭环完整：从“发现与分析”升级为“发现-跟踪-再分析-通知”的持续服务。
- 与市场习惯一致：主流监控产品都把“Alert + Push + Watchlist”作为核心付费能力。

技术合理性：
- 当前仓库已具备 70%+ 核心能力（分析引擎、扫描引擎、推送通道、快照模型）。
- 缺口集中在网关与调度治理，属于可控增量改造，不需要推翻现有架构。

基于当前代码的实现判断（非模板化结论）：
- 已有可直接复用能力：`tools/telegram.py`（仅出站发送）、`agents/scanner_engine.py`（告警分级+快照+触发研究）、`scripts/hourly_watchlist_scan.py`（循环调度骨架）、`agents/workflow_engine.py::run_unified_research`（完整分析）。
- 当前缺失能力：Telegram 入站处理、命令路由、订阅状态持久化、消息重试与 DLQ 状态机。
- 当前依赖约束：`requirements*.txt` 仅包含 `aiohttp/pydantic/streamlit/yfinance` 等基础依赖，未引入 Redis/Celery/APScheduler；因此计划优先给出“标准库 + sqlite 可落地路径”，再给“扩展版”。

风险与边界：
- 实时性依赖数据源稳定性（免费源在峰值时段可能抖动）。
- 数据落地顺序：若当前主要数据源为免费/非流式，先分钟级聚合 + 冷却窗口，再逐步接入 WebSocket 流式源。
- Telegram 平台约束（Webhook 与 getUpdates 互斥、速率限制、消息长度限制）；相关失败/重试状态必须进入 notifications 状态机。
- 合规边界必须明确：仅研究与提醒，不提供自动下单。

2. To-Be 总体架构原则（生产标准）

- 单命令单 `request_id`，单次分析单 `run_id`，全链路可追踪。
- 双层幂等：入站 `update_id` 幂等 + 任务 `request_id` 幂等，避免超时补发/重启恢复导致重复执行。
- 出站幂等：同一触发窗口同一标的去重，避免刷屏。
- 任务持久化：重启后可恢复监控计划与下次触发时间。
- 观察优先：任何失败都结构化记录（source/error_type/backend/message）。
- 通道解耦：Telegram 仅是一个 channel，后续可平滑扩展企业微信/邮件。

3. 路线图（生产版：A/B/C/D）

阶段 A（1-2 周）：Telegram 入站网关与命令协议（P0）
目标：把“只能发消息”升级为“可接收命令并执行分析”。

- A1. Bot Gateway（Webhook + Long Polling 双模式）
  文件建议：`services/telegram_gateway.py`（新增）、`tools/telegram.py`（复用 send 能力）
  完成定义：
  1. 支持 `getUpdates` 与 `setWebhook` 两种部署形态。
  2. `update_id` 幂等消费，异常重试不重复执行。
  3. 支持 `X-Telegram-Bot-Api-Secret-Token` 校验（Webhook）。
  4. `bot_updates.update_id` 建立 unique key，状态机最少包含 `processing/processed/failed/retried`，并记录 `processed_at/error`。
  5. Long polling 模式维护 offset，支持 24 小时窗口内 backlog 补消费；Webhook 模式要求“到达即落库”后再异步处理。

- A2. 命令路由与参数校验
  文件建议：`agents/telegram_command_router.py`（新增）
  完成定义：
  1. 命令：`/analyze <symbol>`、`/monitor <symbol> <interval>`、`/list`、`/stop <job_id|symbol>`、`/help`。
  2. Symbol 规范化复用 `tools.market_data.normalize_market_symbol`。
  3. 错误提示标准化（参数缺失、symbol 无效、权限不足）。
  4. 鉴权策略落地：chat/user 白名单 + 角色权限（尤其 `/monitor` 与 `/stop`）。
  5. 群组权限规则明确：仅管理员可创建/停用订阅，普通成员仅可查询与即时分析。

- A3. 即时分析回推
  文件建议：`services/telegram_actions.py`（新增）
  完成定义：
  1. 路由 `/analyze` 到 `run_unified_research`（直接复用 `agents/workflow_engine.py`，不新建分析内核）。
  2. 超时保护（例如 90s），超时先回执“任务受理”，异步补发结果。
  3. 回执消息必须包含 `request_id`；最终结果消息必须包含 `run_id`，且两者可在存储层关联追踪。
  4. 结果摘要包含：价格、涨跌、RSI、结论、run_id，并附查询命令。
  5. 新增 `analysis_requests(request_id unique, update_id, chat_id, payload, status, run_id, created_at, updated_at, last_error)`，状态最少包含 `queued/running/completed/failed`。
  6. 保证同一 `request_id` 只能进入一次“running/completed”（通过 unique + 状态迁移约束）。

A 验收：
1. Telegram 发 `/analyze AAPL` 可稳定返回结果。  
2. 100 条并发命令下无重复执行（基于 update_id 统计）。  
3. `bot_updates` unique 约束可证明同一 update 仅处理一次。  
4. 网关宕机重启后不丢指令（Long polling offset 或 Webhook 落库恢复可验证）。  
5. 任务幂等 SQL 口径：同一 `request_id` 不得出现多次 `running/completed`。  
6. 并发验收统计口径：`processed_updates == distinct(update_id_received)`，且 `duplicate_update_dropped > 0` 时仍无重复执行。  
7. 以上统计必须进入 `run_report`（作为验收报告字段）。

A 验收 SQL 示例（SQLite/Postgres 通用口径）：
```sql
-- 1) update 幂等：处理数应等于去重后 update_id 数
SELECT COUNT(*) AS processed_updates
FROM bot_updates
WHERE status IN ('processed', 'retried');

SELECT COUNT(DISTINCT update_id) AS distinct_updates
FROM bot_updates;

-- 2) request 幂等：同一 request_id 不能多次进入 running/completed
SELECT request_id, COUNT(*) AS c
FROM analysis_requests
WHERE status IN ('running', 'completed')
GROUP BY request_id
HAVING COUNT(*) > 1;
```


阶段 B（2-4 周）：订阅模型与 24h 调度（P0）
目标：支持“用户在 Telegram 动态创建监控任务，并定时推送”。

- B1. 订阅与任务数据模型
  文件建议：`core/models.py`（扩展）+ `storage/`（新增）
  建议表结构（SQLite/Postgres）：
  1. `telegram_chats(chat_id, user_id, username, created_at, status)`
  2. `watch_jobs(job_id, chat_id, symbol, market, interval_sec, threshold, mode, enabled, next_run_at, created_at, updated_at)`
  3. `watch_events(event_id, job_id, trigger_ts, price, pct_change, reason, rule, bucket_ts, dedupe_key, pushed, run_id)`（`dedupe_key unique` 或 upsert）
  4. `bot_updates(update_id unique, chat_id, command, payload, request_id, processed_at, status, error)`
  5. `notifications(notification_id, event_id, channel, state, retry_count, next_retry_at, last_error, delivered_at)`（`unique(event_id, channel)`）
  6. dedupe 规则标准化：`dedupe_key = job_id + symbol + rule + bucket_ts`（bucket=5min/15min）。
  7. 索引要求：`watch_jobs(chat_id, enabled)`、`watch_jobs(next_run_at)`、`notifications(state, next_retry_at)`。
  8. 灰度控制最小表：`feature_flags(flag_key, enabled)`、`allowlist_chats(chat_id, can_monitor)`。

- B2. 调度引擎
  文件建议：`services/scheduler.py`（新增）
  完成定义：
  1. 支持秒级/分钟级/小时级任务。
  2. v1（当前仓库优先）：复用 `scripts/hourly_watchlist_scan.py` 的 asyncio loop + sqlite job 表；不强依赖新增第三方调度库。
  3. v2（扩展版）：再引入 APScheduler（单实例）或 Celery + Redis（多实例）。
  4. 重启恢复 `enabled=true` 且 `next_run_at<=now` 的任务。
  5. 单实例治理：leader lock/DB lock 防止重复执行。
  6. 多实例治理：job claim 机制（claim TTL + 心跳）避免同一 job 被多 worker 重复跑。

- B3. 监控执行与告警策略
  文件建议：复用 `agents/scanner_engine.py`；新增 `services/watch_executor.py`
  完成定义：
  1. 复用 `ScanConfig`/`run_watchlist_cycle`，对单 symbol/小 watchlist 执行（不改动既有 signal/priority 逻辑）。
  2. `critical` 触发可自动跑 `run_unified_research` 并回推摘要+run_id。
  3. 强制冷却窗口：同 symbol 同类告警 15 分钟内不重复推送。
  4. 重启不丢指令边界：Long polling 依赖 offset+backlog 消费，Webhook 依赖到达落库+幂等键恢复。

B 验收：
1. `/monitor TSLA 1h` 创建任务后可每小时推送。  
2. `/list` 能看到任务状态、下次执行时间、最近触发。  
3. `/stop TSLA` 或 `/stop <job_id>` 可实时停用任务。  
4. 服务重启后任务可自动恢复运行。  
5. 告警风暴下 dedupe 生效：`dedupe_suppressed_count` 上升且推送总量受控。
6. 重启恢复边界验收：Long polling 在 24h backlog 内可补消费且无重复；Webhook 到达落库后可恢复处理且无重复。


阶段 C（4-6 周）：可靠性、风控与成本治理（P1）
目标：从“能跑”提升到“可运营”。

- C1. 限流与配额
  维度：
  1. 每 chat 每分钟命令数限制。
  2. 每 chat 最大监控任务数。
  3. 全局并发上限（防止 LLM/数据 API 爆量）。

- C2. 失败恢复与重试
  规则：
  1. 数据源失败：指数退避重试 + 降级数据源。
  2. Telegram 发送失败：重试队列 + dead-letter（进入 notifications 状态机）。
  3. 队列实现形态：v1 为 sqlite 表驱动队列（轮询 `next_retry_at`），v2 为 Redis/Celery 队列。
  4. 超时任务：回写状态为 `timeout` 并可后台补推。

- C3. 安全基线
  1. Webhook 密钥校验、来源校验、命令白名单。
  2. 脱敏日志（token/chat_id 部分掩码）。
  3. 审计日志只存必要字段，控制 PII 范围。

- C4. 可观测性
  指标：
  1. `command_success_rate`、`monitor_trigger_rate`、`push_success_rate`
  2. `p95_command_latency`、`p95_analysis_latency`
  3. `duplicate_update_dropped`、`dedupe_suppressed_count`
  4. `retry_queue_depth`、`dlq_count`、`notification_state_transition_total`

- C5. 指标触发动作（SLO -> 自动降级）
  规则：
  1. `push_success_rate < 99%` 持续 10 分钟：暂停 monitor 推送，仅保留 `/analyze`。
  2. `dlq_count` 连续 5 分钟上升：切换到摘要短消息模板并降低推送频率。
  3. `p95_analysis_latency` 超阈值持续 10 分钟：暂时关闭 critical 自动 full research，仅推送基础告警。
  4. 自动降级与恢复必须写入状态表（触发时间、触发原因、恢复时间），避免人工口头操作。
  5. 多条件同时触发时优先级：`push_success_rate` 失效 > `dlq_count` 异常 > `latency` 异常（避免冲突策略并发生效）。
  6. 恢复条件：连续 30 分钟 SLI 回归阈值内再自动恢复，恢复事件同样写入状态表。

C 验收：
1. 24h 稳定运行，关键 SLI 达标（例如 push 成功率 > 99%）。  
2. 告警风暴场景下重复推送可控：同 dedupe_key 在冷却窗口内最多推送 1 次。  
3. 能按 chat/job/source 定位错误。
4. 降级规则可自动触发并可回滚恢复（有状态记录）。

5. 持续质量门禁（生产版）

- 单测：命令解析、参数校验、调度/恢复、推送幂等。
- 集成：Webhook/LongPolling 二选一端到端；`/analyze` 与 `/monitor` 回归。
- 压测：并发命令、告警风暴、Telegram API 失败注入。
- 强制验收产物：每日生成 `run_report`（含 SQL 口径统计 + 指标快照）。
- `run_report` 最低指标集合：
  1. `command_success_rate`, `push_success_rate`
  2. `duplicate_update_dropped`, `dedupe_suppressed_count`
  3. `retry_queue_depth`, `dlq_count`
  4. `p95_command_latency`, `p95_analysis_latency`
- 运维门禁：
  1. `command_success_rate`、`push_success_rate` 达标后再放量。
  2. 出现持续失败自动降级为“只分析不推送”或“只推送摘要”。
  3. 明确刷屏判定：同 `dedupe_key` 超过 1 次/冷却窗口视为失败。

6. 市场产品做法综合（可直接借鉴）

- 做法一：把“条件触发”放在图表/监控平台，把“执行与分发”放在后端网关。  
  代表：TradingView、TrendSpider（Webhook 模式）。

- 做法二：把“监控任务”产品化为可管理对象（列表、启停、频率、通知偏好）。  
  代表：Koyfin Alerts Center、Freqtrade Telegram 命令集。

- 做法三：把“命令式查询 + 周期性提醒”并存。  
  代表：CoinMarketCap Telegram Bot（即时查询 + 小时提醒）。

- 做法四：在数据层用流式接口，减少轮询与延迟。  
  代表：Alpaca/Polygon WebSocket 数据流。

对 Alpha-Insight 的建议：  
优先落地“命令层 + 订阅层 + 调度层”，先实现高确定性的 `/analyze` 与 `/monitor`；后续再升级复杂策略（多条件、组合监控、跨市场联动）。
